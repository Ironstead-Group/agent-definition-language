# OpenClaw Passport Example

Reference implementation for the verification procedure proposed in
[`proposals/2026-05-03-passport-verification-procedure.md`](../../../../proposals/2026-05-03-passport-verification-procedure.md).

Three scenarios demonstrate ADL passport identity, discovery, and verification
end-to-end:

1. **Consumer setup** вҖ” a Mac Mini-style local OpenClaw host provisions an
   agent passport with cryptographic identity, signs it, and presents it
   when calling other agents.
2. **Enterprise discovery** вҖ” a company onboards multiple agents and serves
   them via `/.well-known/adl-agents` (spec В§6.4) so peers can resolve them.
3. **Platform validation plugin** вҖ” the OpenClaw gateway plugin runs the
   full В§10.3 verification procedure on every inbound request and either
   allows or rejects based on the structured outcome.

## Run the demo

```bash
bun run packages/adl-agent/examples/openclaw-passport/run-demo.ts
```

The orchestrator runs all three phases, then exercises four verification
scenarios (valid, tampered, expired, retired) so each `MUST` step in the
procedure is demonstrated under both pass and fail conditions.

### Verification primitives live in `@adl-spec/core`

The verification procedure (В§10.3) and supporting primitives вҖ” Ed25519
signing/verifying, JCS canonicalization, `did:web` resolution, discovery
client, programmatic passport construction вҖ” are exposed from
[`@adl-spec/core`](../../../adl-core/src/verify/) as a public API:

```ts
import {
  // В§10.3 procedure
  verifyPassport,
  type VerifyConfig,
  type VerificationOutcome,
  DEFAULT_VERIFY_CONFIG,

  // Primitives
  generateKeyPair,
  signCanonical,
  verifyCanonical,
  jcsCanonicalize,

  // did:web
  resolveDIDWeb,
  buildDIDDocument,

  // Discovery
  fetchDiscoveryDocument,
  fetchPassportFromUrl,
  discoverAndFetchAll,

  // Construction
  buildPassport,
  signPassport,
} from "@adl-spec/core";
```

Adapters for other agent frameworks (A2A middleware, Google ADK callbacks,
MCP server hooks, LangGraph nodes) call the same functions вҖ” only the
glue that extracts the passport and translates the outcome differs per
framework.

## Verification scenarios

The orchestrator presents four passport variants to the validator. Expected
outcomes:

| Scenario | Modification | Step that blocks | Spec anchor |
|----------|--------------|------------------|-------------|
| A. Valid passport | none | none вҖ” passes | В§10.3.1.1вҖ“В§10.3.1.9 |
| B. Tampered passport | add `evil.example` to `allowed_hosts` after signing | Signature verification fails | В§10.3.1.5 |
| C. Expired attestation | re-sign with `expires_at` in 2024 | Schema validation fails (ADL-4003) | В§10.3.1.2 |
| D. Retired requesting agent | re-sign with `lifecycle.status: retired` | Lifecycle gating fails | В§10.3.1.7 |

Note on Scenario C: The schema validator in `@adl-spec/core` already enforces
attestation expiry via error code ADL-4003. The verification procedure's В§10.3.1.6
temporal check is a defense-in-depth backstop for cases where schema validation
is run with relaxed options or skipped. Both surfaces correctly block the same
input вҖ” the demo shows the earlier (cheaper) check firing first.

## Implementation notes

### Trust anchor and the chicken-and-egg problem

A self-signed passport that includes its own public key is not, on its own,
trustworthy: anyone can generate a key, claim any identity, and sign. The
example resolves this two ways:

1. **`did:web` resolution** вҖ” the DID Document is published at a URL
   controlled by the agent's domain authority. The TLS authority is the
   trust anchor; the inline `cryptographic_identity.public_key` is
   cross-checked against the resolved key (В§10.3.1.4).
2. **Trust-on-First-Use** вҖ” for consumer setups without a real domain, the
   validator can be configured to accept the inline key the first time it
   sees the agent. Demo configuration uses `trustOnFirstUse: true` so the
   consumer/Mac-Mini scenario works without provisioning a real
   `home.local` DNS record.

Production consumer deployments would either (a) own a real domain and
publish a DID Document, or (b) use third-party attestation, in which case
the verifier trusts the issuer's well-known key and the issuer signs the
passport.

### TLS bypass for localhost

В§10.3.1.1 requires HTTPS retrieval. The demo runs entirely on localhost so
TLS is not provisioned. The retrieval-integrity step recognizes
`localhost:NNNN` authorities and passes with a warning. Production
implementations MUST require HTTPS.

### Enterprise key provisioning

The enterprise gateway provisions a fresh Ed25519 keypair for each agent on
every start. This is fine for a demo but means signatures change every run.
Production deployments would source keys from a KMS / HSM with durable
storage and rotate them through `attestation.issued_at` updates.

### Strict validation modes

Set `VALIDATOR_MODE=audit` or `VALIDATOR_MODE=permissive` to see the other
two enforcement modes from the OpenClaw runtime layer proposal:

```bash
VALIDATOR_MODE=audit bun run packages/adl-agent/examples/openclaw-passport/run-demo.ts
```

Audit mode passes failing requests through but logs the structured outcome.
Permissive mode does the same and attaches `ADL-Validation` headers to
the upstream response.
